# 평가와 지표

## 이 문서에서 배우는 것
- AI 시스템을 어떻게 평가하는지 기본 틀을 이해한다.
- 정확도, 정밀도, 재현율, F1, perplexity, retrieval quality의 의미를 구분한다.
- 모델 성능과 사용자 체감 품질이 항상 같지 않다는 점을 본다.

## 왜 중요한가
모델을 만들거나 붙이는 것보다 더 어려운 일이 "잘 되고 있는지 판단하는 것"이다. 평가가 없으면 개선도 없다. 특히 LLM과 RAG는 정답이 애매한 경우가 많아, 어떤 지표를 봐야 하는지 감이 중요하다.

## 핵심 개념 설명
분류 문제에서는 정확도만 보면 놓치는 것이 많다. 정밀도는 양성이라고 예측한 것 중 실제 양성 비율이고, 재현율은 실제 양성 중 얼마나 잘 찾아냈는지다. F1은 둘의 균형을 본다. 언어 모델에서는 perplexity가 "다음 토큰 예측이 얼마나 헷갈리는가"를 나타내는 지표로 쓰인다. RAG에서는 검색 품질이 매우 중요해서 관련 문서를 잘 찾았는지 따로 봐야 한다.

핵심 지표:

| 지표 | 무엇을 보나 | 주의점 |
| --- | --- | --- |
| 정확도 | 전체 중 맞춘 비율 | 클래스 불균형에 약함 |
| 정밀도 | 맞다고 한 것의 정확함 | 놓친 양성은 반영 약함 |
| 재현율 | 실제 양성을 얼마나 찾았나 | 오탐 증가 가능 |
| F1 | 정밀도와 재현율 균형 | 상황별 해석 필요 |
| Perplexity | 언어 예측의 불확실성 | 실제 사용자 품질과 1:1은 아님 |
| Retrieval Quality | 검색 문서의 관련성 | 생성 품질과 함께 봐야 함 |

RAG 평가는 보통 두 단계로 본다.
1. 검색이 맞는 문서를 가져왔는가
2. 생성이 그 문서를 근거로 잘 답했는가

## 예시 / 비유
의료 진단처럼 놓치면 큰일 나는 문제는 재현율이 중요할 수 있다. 반대로 스팸 차단처럼 정상 메일을 잘못 막으면 불편한 문제는 정밀도가 더 중요할 수 있다.

## 한눈에 정리
- 하나의 지표만 보면 위험하다.
- 분류, 생성, 검색은 평가 방식이 다르다.
- RAG는 검색 품질과 생성 품질을 분리해서 봐야 한다.

## 자주 하는 오해
- 정확도가 높으면 좋은 모델이라고 단정하면 안 된다.
- perplexity가 낮다고 항상 사용자 만족도가 높은 것은 아니다.
- LLM 평가는 감으로만 하면 안 되고, 기준과 샘플셋이 필요하다.

## 체크포인트
- 정밀도와 재현율의 차이를 설명할 수 있는가?
- 정확도만으로 부족한 상황을 예로 들 수 있는가?
- RAG 평가에서 검색과 생성을 분리해 봐야 하는 이유를 이해했는가?

## 한 줄 요약
AI 평가는 문제 유형에 맞는 지표를 고르고, 특히 LLM과 RAG는 여러 층위로 나눠서 봐야 한다.
