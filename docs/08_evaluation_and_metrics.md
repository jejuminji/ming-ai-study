# 평가와 지표

## 이 문서에서 배우는 것
- AI 시스템을 어떻게 평가하는지 기본 틀을 이해한다.
- 정확도, 정밀도, 재현율, F1, perplexity, retrieval quality의 의미를 구분한다.
- 모델 성능과 사용자 체감 품질이 항상 같지 않다는 점을 본다.
- LLM과 RAG를 어떻게 나눠 평가할지 감을 잡는다.

## 왜 중요한가
모델을 만들거나 붙이는 것보다 더 어려운 일이 "잘 되고 있는지 판단하는 것"이다. 평가가 없으면 개선도 없다. 특히 LLM과 RAG는 정답이 애매한 경우가 많아, 어떤 지표를 봐야 하는지 감이 중요하다.

실무에서는 모델을 조금 바꿨을 때 좋아졌는지 나빠졌는지를 정량적으로 확인할 수 있어야 한다. 그래야 프롬프트 변경, 검색기 교체, 모델 업그레이드, 파라미터 조정의 효과를 비교할 수 있다.

## 핵심 개념 설명
분류 문제에서는 정확도만 보면 놓치는 것이 많다. 정밀도는 양성이라고 예측한 것 중 실제 양성 비율이고, 재현율은 실제 양성 중 얼마나 잘 찾아냈는지다. F1은 둘의 균형을 본다. 언어 모델에서는 perplexity가 "다음 토큰 예측이 얼마나 헷갈리는가"를 나타내는 지표로 쓰인다. RAG에서는 검색 품질이 매우 중요해서 관련 문서를 잘 찾았는지 따로 봐야 한다.

핵심 지표:

| 지표 | 무엇을 보나 | 주의점 | 자주 쓰는 상황 |
| --- | --- | --- | --- |
| 정확도 | 전체 중 맞춘 비율 | 클래스 불균형에 약함 | 일반 분류 |
| 정밀도 | 맞다고 한 것의 정확함 | 놓친 양성은 반영 약함 | 스팸 탐지, 경고 시스템 |
| 재현율 | 실제 양성을 얼마나 찾았나 | 오탐 증가 가능 | 의료, 이상 탐지 |
| F1 | 정밀도와 재현율 균형 | 상황별 해석 필요 | 불균형 분류 |
| Perplexity | 언어 예측의 불확실성 | 실제 사용자 품질과 1:1은 아님 | 언어 모델 연구/비교 |
| Retrieval Quality | 검색 문서의 관련성 | 생성 품질과 함께 봐야 함 | RAG |

## 작은 사례로 보기

### 사례 1: 스팸 분류
메일 100개 중 95개를 맞췄다면 정확도는 95%일 수 있다. 하지만 스팸이 5개뿐이고 그중 4개를 놓쳤다면 실제 서비스 관점에서는 매우 안 좋을 수 있다. 그래서 정확도만 보면 위험하다.

### 사례 2: 병원 이상 징후 탐지
정밀도보다 재현율이 더 중요할 수 있다. 잘못된 경고가 좀 늘어나더라도, 진짜 위험 신호를 놓치지 않는 편이 더 중요하기 때문이다.

### 사례 3: RAG 챗봇
답변이 틀렸을 때 원인이 둘 중 어디에 있는지 나눠 봐야 한다.
1. 검색이 엉뚱한 문서를 가져왔는가
2. 문서는 맞았는데 생성이 잘못 요약했는가

## LLM과 RAG 평가의 차이
LLM 평가는 보통 아래처럼 나눠 보는 편이 좋다.
- 형식 준수: 원하는 포맷을 지키는가
- 사실성: 근거 없는 내용을 지어내지 않는가
- 유용성: 사용자가 원하는 작업을 실제로 돕는가
- 일관성: 비슷한 입력에 비슷한 품질을 내는가

RAG 평가는 보통 두 단계로 본다.
1. 검색이 맞는 문서를 가져왔는가
2. 생성이 그 문서를 근거로 잘 답했는가

즉, RAG 품질은 "검색 + 생성"의 합이다.

## 예시 / 비유
의료 진단처럼 놓치면 큰일 나는 문제는 재현율이 중요할 수 있다. 반대로 스팸 차단처럼 정상 메일을 잘못 막으면 불편한 문제는 정밀도가 더 중요할 수 있다.

RAG는 시험 문제를 푸는 학생과 비슷하다. 참고서를 엉뚱하게 가져오면 답이 틀릴 가능성이 높고, 참고서를 잘 가져와도 잘못 읽으면 역시 답이 틀린다.

## 평가를 설계할 때 묻는 질문
- 이 문제에서 가장 치명적인 실패는 무엇인가?
- 놓치는 것이 더 위험한가, 잘못 맞히는 것이 더 위험한가?
- 자동 지표만으로 충분한가, 사람 평가도 필요한가?
- 모델 문제가 아니라 데이터/검색/프롬프트 문제는 아닌가?

## 한눈에 정리
- 하나의 지표만 보면 위험하다.
- 분류, 생성, 검색은 평가 방식이 다르다.
- RAG는 검색 품질과 생성 품질을 분리해서 봐야 한다.
- 사용자 만족도와 모델 점수는 항상 같지 않다.

## 자주 하는 오해
- 정확도가 높으면 좋은 모델이라고 단정하면 안 된다.
- perplexity가 낮다고 항상 사용자 만족도가 높은 것은 아니다.
- LLM 평가는 감으로만 하면 안 되고, 기준과 샘플셋이 필요하다.
- RAG 오류를 전부 모델 탓으로 돌리면 문제 원인을 놓치기 쉽다.

## 체크포인트
- 정밀도와 재현율의 차이를 설명할 수 있는가?
- 정확도만으로 부족한 상황을 예로 들 수 있는가?
- RAG 평가에서 검색과 생성을 분리해 봐야 하는 이유를 이해했는가?
- 내가 다루는 제품에서 가장 중요한 실패 유형이 무엇인지 말할 수 있는가?

## 한 줄 요약
AI 평가는 문제 유형에 맞는 지표를 고르고, 특히 LLM과 RAG는 검색·생성·사용자 체감 품질을 나눠서 봐야 한다.
