# AI 시스템 설계 기초

## 이 문서에서 배우는 것
- AI 기능을 제품으로 만들 때 모델 외에 무엇을 고려해야 하는지 이해한다.
- 입력, 모델, 검색, 메모리, 평가, 모니터링이 어떻게 연결되는지 본다.
- "좋은 모델"과 "좋은 시스템"이 다른 문제라는 점을 안다.
- 비용, 속도, 안정성, 안전성을 함께 보는 관점을 익힌다.

## 왜 중요한가
실무에서는 모델 하나를 잘 고르는 것만으로는 충분하지 않다. 비용, 속도, 안정성, 보안, 데이터 흐름, 관찰 가능성까지 함께 설계해야 실제 서비스가 된다. 특히 LLM 기반 기능은 프롬프트, 검색, 캐시, 가드레일 같은 주변 요소의 영향이 크다.

사용자는 "모델이 몇 파라미터인지"보다 "잘 대답하는지", "빨리 뜨는지", "틀려도 위험하지 않은지"를 더 크게 체감한다. 그래서 시스템 설계는 모델 선택보다 넓은 문제다.

## 핵심 개념 설명
AI 시스템은 보통 다음 흐름으로 본다. 사용자 입력이 들어오면 필요한 전처리를 하고, 모델 또는 검색 시스템을 호출해 결과를 만든 뒤, 후처리와 검증을 거쳐 사용자에게 응답한다. 이 전 과정에서 로그, 평가, 모니터링이 따라붙어야 한다.

핵심 구성 요소:

| 요소 | 역할 | 실무 질문 |
| --- | --- | --- |
| 입력 처리 | 사용자 요청 정리 | 어떤 정보가 빠졌나? |
| 모델 계층 | 생성 또는 예측 | 어떤 모델이 비용 대비 적절한가? |
| 검색 계층 | 외부 지식 연결 | 어떤 문서를 얼마나 잘 찾나? |
| 메모리 | 상태와 이력 관리 | 무엇을 저장하고 얼마나 유지할까? |
| 가드레일 | 안전성 확보 | 금지 응답, 민감 정보는 어떻게 막나? |
| 평가/모니터링 | 품질 추적 | 성능 저하를 어떻게 감지하나? |

에이전트형 시스템에서는 계획과 도구 호출 실패 처리도 중요하다. 한 단계 실패가 전체 실패로 이어지지 않게 재시도와 fallback 전략이 필요하다.

## 작은 서비스 예시
"회사 문서를 바탕으로 답하는 내부 챗봇"을 만든다고 생각해 보자.
1. 사용자가 질문을 입력한다.
2. 시스템이 질문을 정리하고 필요하면 검색어를 만든다.
3. 검색 계층이 관련 문서를 찾는다.
4. 모델이 문서와 질문을 함께 보고 답변을 만든다.
5. 후처리에서 출처를 붙이고 금지 표현을 검사한다.
6. 로그를 남기고, 나중에 품질 평가에 활용한다.

이 흐름 어디에서든 문제가 생길 수 있다. 검색이 틀릴 수도 있고, 모델이 문서를 잘못 읽을 수도 있고, 응답이 너무 느릴 수도 있다. 그래서 시스템은 "한 부품"이 아니라 "연결된 체인"으로 봐야 한다.

## 예시 / 비유
좋은 주방이 좋은 칼 하나로 완성되지 않는 것처럼, AI 시스템도 좋은 모델 하나로 완성되지 않는다. 재료 관리, 조리 순서, 위생, 서빙, 피드백까지 있어야 제대로 운영된다.

현실적인 대응 관계로 보면:
- 모델 = 요리사
- 검색 = 재료 창고
- 가드레일 = 위생 규칙
- 모니터링 = 매장 운영 점검
- 캐시/최적화 = 바쁜 시간대 운영 전략

## 설계할 때 자주 보는 축

### 품질
정확한가, 근거가 있는가, 사용자 의도에 맞는가를 본다.

### 속도
응답이 너무 느리면 품질이 좋아도 사용성이 떨어진다.

### 비용
큰 모델, 긴 컨텍스트, 잦은 검색은 비용 증가로 이어질 수 있다.

### 안정성
외부 API 실패, 검색 실패, 타임아웃이 나도 시스템이 완전히 무너지지 않아야 한다.

### 안전성
민감 정보 노출, 허용되지 않은 요청, 위험한 출력에 대한 방어가 필요하다.

## 실무에서 자주 하는 개선
- 검색 품질을 높이기 위해 청크 크기와 인덱싱 방식을 조정한다.
- 응답 속도를 줄이기 위해 캐시나 더 작은 모델을 일부 경로에 쓴다.
- 품질 문제를 추적하기 위해 로그와 평가셋을 만든다.
- 실패 시 더 단순한 fallback 응답을 준비한다.

## 한눈에 정리
- 모델은 시스템의 한 부품이다.
- 검색, 메모리, 가드레일, 관찰 가능성이 함께 필요하다.
- 실무에서는 정확도뿐 아니라 비용과 지연 시간도 핵심이다.
- 좋은 AI 제품은 "모델 성능 + 시스템 운영력"의 결과다.

## 자주 하는 오해
- 최신 모델만 쓰면 시스템 품질이 자동으로 좋아지는 것은 아니다.
- 프롬프트는 설계의 전부가 아니다.
- 운영 단계의 로그와 평가 없이 안정적인 개선은 어렵다.
- 프로토타입이 잘 되면 운영 환경에서도 그대로 잘 될 것이라고 생각하기 쉽다.

## 체크포인트
- AI 시스템에서 모델 외에 중요한 요소 3가지를 말할 수 있는가?
- RAG 시스템에서 검색 품질이 왜 전체 품질에 큰 영향을 주는지 설명할 수 있는가?
- 비용, 속도, 품질이 함께 고려되어야 한다는 점을 이해했는가?
- 문제가 생겼을 때 모델 문제와 시스템 문제를 나눠 생각할 수 있는가?

## 한 줄 요약
AI 시스템 설계는 모델 선택보다 넓은 문제이며, 검색·메모리·안전성·모니터링·비용까지 함께 설계해야 실제 서비스가 된다.
