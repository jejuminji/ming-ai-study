# 프롬프팅, RAG, 에이전트

## 이 문서에서 배우는 것
- 프롬프팅, RAG, 에이전트가 각각 무엇을 해결하는지 이해한다.
- 세 개념이 경쟁 관계가 아니라 서로 다른 레이어라는 점을 본다.
- 툴 사용, 메모리, 계획, 실행 흐름 관점에서 에이전트를 이해한다.

## 왜 중요한가
실무에서는 "LLM을 쓴다"보다 "LLM을 어떻게 시스템에 붙이느냐"가 더 중요하다. 같은 모델이라도 프롬프트만 바꿀지, 외부 지식을 붙일지, 도구를 쓰게 할지에 따라 제품 성격이 크게 달라진다.

## 핵심 개념 설명
프롬프팅은 모델에게 지시를 더 잘 전달하는 방법이다. RAG는 검색 기반으로 외부 문서를 가져와 답변에 참고하게 하는 구조다. 에이전트는 모델이 계획을 세우고, 도구를 호출하고, 중간 결과를 바탕으로 다음 행동을 결정하는 시스템이다.

비교하면:

| 개념 | 해결하려는 문제 | 핵심 메커니즘 |
| --- | --- | --- |
| 프롬프팅 | 같은 모델에서 더 나은 출력 얻기 | 지시, 맥락, 예시 설계 |
| RAG | 모델 밖의 지식 활용 | 검색 + 문맥 주입 |
| 에이전트 | 다단계 작업 자동화 | 계획 + 도구 사용 + 반복 실행 |

에이전트를 볼 때는 네 가지 축이 중요하다.
- 툴 사용: 검색, 계산기, 코드 실행기, API 호출 등
- 메모리: 대화 기록, 작업 상태, 사용자 선호 저장
- 계획: 해야 할 일을 단계로 나누기
- 실행 흐름: 관찰 → 판단 → 행동 → 결과 확인 반복

## 예시 / 비유
프롬프팅은 사람에게 일을 잘 부탁하는 방식과 비슷하다. RAG는 회의 전에 관련 문서를 찾아 책상 위에 올려두는 일에 가깝다. 에이전트는 문서를 읽고, 캘린더를 확인하고, 메일을 보내는 비서 시스템에 더 가깝다.

## 한눈에 정리
- 프롬프팅은 입력 설계
- RAG는 지식 연결
- 에이전트는 행동 orchestration

## 자주 하는 오해
- 프롬프트만 잘 쓰면 모든 문제가 해결되지는 않는다.
- RAG는 단순 검색이 아니라 검색 결과를 생성과 연결하는 구조다.
- 에이전트는 "모델 하나"가 아니라 여러 구성 요소를 묶은 시스템이다.

## 체크포인트
- 프롬프팅과 RAG의 차이를 설명할 수 있는가?
- 에이전트를 툴 사용과 실행 흐름 관점에서 설명할 수 있는가?
- 최신 지식이 필요한 경우 왜 RAG가 자주 쓰이는지 이해했는가?

## 한 줄 요약
프롬프팅은 지시를 개선하고, RAG는 외부 지식을 붙이며, 에이전트는 모델이 도구를 써서 여러 단계를 수행하게 만든다.
